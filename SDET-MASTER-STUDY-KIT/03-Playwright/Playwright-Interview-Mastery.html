Playwright: cross-browser, auto-waiting, network interception, native mobile emulation. Selenium: broader language support, legacy enterprise adoption, slower. What are Browser Contexts and why are they efficient?IntermediateP1
Basic Answer

Browser Contexts are isolated sessions within a single browser instance (~50ms creation vs ~500ms+ for new browser). They share the browser process but have separate storage, cookies, permissions, and localStorage. Enables parallel execution with complete isolation using a fraction of the memory.

How does auto-waiting work in Playwright vs Selenium's explicit waits?AdvancedP1
Basic Answer

Playwright automatically waits for actionability — visibility, stability, enabled state, no attached listeners. Built into every action and assertion. Selenium requires explicit WebDriverWait calls. Never use page.waitForTimeout() — it's an anti-pattern. Use web-first assertions instead.

Basic Answer

How do web-first assertions work and how are they different from standard assertions?IntermediateP1
Basic Answer

Web-first assertions (toBeVisible, toHaveText) auto-retry until condition met or timeout expires. Standard assertions (expect(value).toBe()) evaluate immediately. Use expect.poll() or expect.toPass() for custom retrying assertions.

How does Playwright's fixture system enable dependency injection? Design a multi-layer fixture chain for enterprise testing.ExpertP1
Basic Answer

Fixtures in Playwright are a dependency injection system. Each fixture can depend on other fixtures, and Playwright resolves the dependency graph automatically. A multi-layer chain: config → auth → db → apiClient → page → authenticatedPage. Fixtures have scope: 'test' (per test) or scope: 'worker' (per worker). The use() callback provides setup AND teardown in one construct.

Detailed Answer
import { test as base } from '@playwright/test'; // Layer 1: Config fixture (worker-scoped, loaded once) const test = base.extend<{ config: TestConfig; db: Database; api: ApiClient; authPage: Page }>({ config: [async ({}, use) => { const config = await loadConfig(); await use(config); }, { scope: 'worker' }], // Layer 2: DB fixture depends on config db: [async ({ config }, use) => { const db = await Database.connect(config.dbUrl); await db.seedTestData(); await use(db); await db.cleanup(); // Auto-teardown }, { scope: 'worker' }], // Layer 3: API client depends on config api: [async ({ config }, use) => { const api = new ApiClient(config.baseUrl); await api.login(config.adminUser); await use(api); }, { scope: 'worker' }], // Layer 4: Authenticated page — depends on page + api authPage: [async ({ page, api }, use) => { const token = await api.getToken(); await page.goto('/login'); await page.evaluate(`localStorage.setItem('token', '${token}')`); await page.goto('/dashboard'); await use(page); }, { scope: 'test' }] }); // Test uses only the fixtures it needs test('user dashboard loads', async ({ authPage, db }) => { await expect(authPage.locator('h1')).toHaveText('Dashboard'); });
⚠️ Optimization Worker-scoped fixtures reduce setup overhead. A DB connection per worker (~5-10s) vs per test (~5-10ms context creation). Use scope: 'worker' for expensive setup, scope: 'test' for test-specific state.
Real Interview Answer

"I design fixture chains based on scope and dependency. Worker-scoped fixtures handle expensive setup like DB connections, API tokens, and config loading — once per worker. Test-scoped fixtures handle per-test state like authenticated pages with unique user credentials. The key insight is that fixtures auto-resolve dependencies, provide built-in teardown, and eliminate the need for beforeEach/afterEach hooks. In an enterprise framework with 30+ microservices, this pattern reduced our test setup time by 60%."

Design a custom Playwright reporter. How would you stream real-time results to a custom dashboard?ExpertP1
Basic Answer

Implement the Reporter interface from @playwright/test/reporter. Key methods: onBegin(), onTestBegin(), onTestEnd(), onEnd(), onStepBegin(), onStepEnd(), onError(). Each receives detailed test result objects with status, duration, errors, attachments, and annotations. Stream results via WebSocket, HTTP POST, or write to a message queue.

Detailed Answer
import { Reporter, TestCase, TestResult, FullConfig, Suite } from '@playwright/test/reporter'; class CustomDashboardReporter implements Reporter { private apiUrl: string; private results: TestResult[] = []; onBegin(config: FullConfig, suite: Suite) { this.apiUrl = process.env.DASHBOARD_URL || 'http://localhost:3000/api/results'; console.log(`Starting suite: ${suite.title}`); } onTestEnd(test: TestCase, result: TestResult) { const payload = { title: test.title, status: result.status, duration: result.duration, errors: result.errors.map(e => e.message), retry: result.retry, project: test.parent.project()?.name, tags: test.tags, timestamp: new Date().toISOString() }; // Non-blocking fire-and-forget POST fetch(this.apiUrl, { method: 'POST', body: JSON.stringify(payload), headers: { 'Content-Type': 'application/json' } }).catch(() => {}); // Don't let reporting failure affect tests } onEnd() { const summary = { total: this.results.length, passed: this.results.filter(r => r.status === 'passed').length, failed: this.results.filter(r => r.status === 'failed').length, duration: this.results.reduce((sum, r) => sum + r.duration, 0) }; console.log(`Suite complete: ${JSON.stringify(summary)}`); } } export default CustomDashboardReporter;

Config in playwright.config.ts: reporter: [['html'], ['./CustomDashboardReporter.ts']]. Multiple reporters run simultaneously.

How do you implement enterprise-grade authentication patterns with Playwright? Compare storageState, API injection, and cookie manipulation.AdvancedP1
Basic Answer

Three patterns: (1) storageState: Login once via UI in globalSetup, save cookies/localStorage, reuse across tests. Fastest (~50ms per test). (2) API token injection: Get token via API call, set in localStorage or cookies via page.evaluate(). No UI login needed. (3) Cookie manipulation: Direct cookie setting via context.addCookies(). StorageState is preferred for most scenarios; API injection for stateless/SPA apps; cookie manipulation for legacy apps.

Detailed Answer

Enterprise auth — multi-role strategy:

// global-setup.ts — authenticate multiple roles once async function globalSetup(config: FullConfig) { const { browser } = await chromium.launch(); const roles = ['admin', 'user', 'readonly']; for (const role of roles) { const page = await browser.newPage(); await page.goto(process.env.BASE_URL!); await page.fill('#username', process.env[`${role.toUpperCase()}_USER`]!); await page.fill('#password', process.env[`${role.toUpperCase()}_PASS`]!); await page.click('#login'); await page.waitForURL('/dashboard'); await page.context().storageState({ path: `${role}-auth.json` }); await page.close(); } await browser.close(); } // playwright.config.ts — per-role projects projects: [ { name: 'admin-chrome', use: { storageState: 'admin-auth.json' } }, { name: 'user-chrome', use: { storageState: 'user-auth.json' } }, ]
⚠️ Security Note Never commit auth JSON files. Generate them in CI via globalSetup with secrets from environment variables. Use .gitignore patterns and ephemeral test credentials that expire after each CI run.

How do you implement self-healing locators in Playwright? What strategies work at enterprise scale?ExpertP1
Basic Answer

Self-healing locators automatically fall back to alternative selectors when the primary locator fails. Strategies: (1) Multi-locator: Chain multiple locators using .or(): page.locator('button').or(page.locator('[role=button]')). (2) Custom fixture: Wrap Playwright locator with fallback logic. (3) AI-powered: Use Playwright's Codegen/Inspector to regenerate locators. (4) Attribute hierarchy: data-testid → role → label → text → CSS. Best practice: prefer resilient locators (getByRole) over self-healing — prevention > cure.

Detailed Answer
// Custom resilient locator with fallback chain async function resilientLocator(page: Page, selectors: string[]) { for (const selector of selectors) { const locator = page.locator(selector); try { await locator.waitFor({ state: 'visible', timeout: 2000 }); return locator; } catch { console.warn(`Selector '${selector}' not found, trying next...`); } } throw new Error(`None of the selectors found: ${selectors.join(', ')}`); } // Usage const submitBtn = await resilientLocator(page, [ '[data-testid="submit-btn"]', 'button:has-text("Submit")', '[role="button"]:has-text("Submit")', '//button[contains(text(),"Submit")]' ]);
⚠️ Tradeoff Self-healing increases test execution time (each fallback adds a timeout). Use only for known volatile selectors. Prefer getByRole and getByTestId which rarely need healing.

How do you integrate OWASP ZAP DAST scanning with Playwright for automated security testing in CI/CD?ExpertP3
Basic Answer

OWASP ZAP is the industry-standard DAST (Dynamic Application Security Testing) tool. Integration pattern: (1) Configure Playwright to route traffic through ZAP proxy (--proxy-server=http://localhost:8080). (2) Perform authenticated UI flows using Playwright — ZAP passively spiders all pages visited. (3) Inject Playwright's auth state into ZAP so the active scan stays authenticated. (4) Trigger ZAP's Active Scan API (http://zap-api/json/ascan/action/scan/). (5) Poll scan progress and retrieve alerts. (6) Assert against alert severity thresholds in CI — fail build on High/Critical alerts.

Detailed Answer

End-to-end security testing pipeline:

// playwright.config.ts — proxy configuration export default defineConfig({ use: { proxy: { server: 'http://localhost:8080' }, // ZAP proxy storageState: 'auth.json', }, globalSetup: './security.setup.ts', }); // security.setup.ts — launch ZAP and authenticate import { chromium } from '@playwright/test'; import ZAPClient from 'zaproxy'; async function securityGlobalSetup() { // Start ZAP daemon const zap = new ZAPClient({ apiKey: 'your-api-key', proxy: { host: 'localhost', port: 8080 } }); // Use Playwright to authenticate and let ZAP spider const browser = await chromium.launch(); const context = await browser.newContext(); const page = await context.newPage(); await page.goto(process.env.BASE_URL!); await page.fill('#username', process.env.AUTH_USER!); await page.fill('#password', process.env.AUTH_PASS!); await page.click('#login'); await page.waitForURL('**/dashboard'); // Navigate through key pages for ZAP to spider const pagesToScan = ['/users', '/settings', '/api/docs', '/profile']; for (const path of pagesToScan) { await page.goto(path); await page.waitForLoadState('networkidle'); } // Trigger ZAP Active Scan asynchronously const scanId = await zap.ascan.scan( process.env.BASE_URL!, true, // recurse false, // inScopeOnly '', // scanPolicyName '', // method 'POST,GET,PUT,DELETE,PATCH' // httpMethods ); // Poll until scan completes let progress = 0; while (progress < 100) { progress = await zap.ascan.status(scanId); await new Promise(r => setTimeout(r, 2000)); } // Get alerts and assert const alerts = await zap.core.alerts(process.env.BASE_URL!); const highRisk = alerts.filter(a => a.risk === 'High'); const mediumRisk = alerts.filter(a => a.risk === 'Medium'); console.log(`Security scan complete: ${highRisk.length} High, ${mediumRisk.length} Medium`); // Fail build on High/Critical — never deploy with known vulns if (highRisk.length > 0) { console.error('❌ BUILD BLOCKED: High-risk vulnerabilities found!'); process.exit(1); } await browser.close(); await zap.core.shutdown(); } export default securityGlobalSetup;
