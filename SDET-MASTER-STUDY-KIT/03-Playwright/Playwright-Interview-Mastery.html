What are Browser Contexts and why are they efficient?IntermediateP1
Basic Answer

Browser Contexts are isolated sessions inside one browser instance. They take ~50ms to create (vs ~500ms+ for a new browser). They share the browser process but have separate cookies, storage, and permissions. This lets you run tests in parallel with full isolation while using much less memory.

Code Example
import { chromium } from 'playwright'; async function contextDemo() { const browser = await chromium.launch(); // Context 1 — logged in as admin (takes ~50ms to create) const adminContext = await browser.newContext(); const adminPage = await adminContext.newPage(); await adminPage.goto('https://example.com'); // adminContext has its own cookies, localStorage, session // Context 2 — logged in as regular user, fully isolated const userContext = await browser.newContext(); const userPage = await userContext.newPage(); await userPage.goto('https://example.com'); // No cookies shared with adminContext — complete isolation! // Both contexts share the same browser process (saves ~500ms each) await browser.close(); }

How does auto-waiting work in Playwright vs Selenium's explicit waits?AdvancedP1
Basic Answer

Playwright automatically waits before doing anything — it checks that elements are visible, stable, enabled, and not blocked by other elements. This is built into every action and assertion. Selenium makes you write WebDriverWait manually. NEVER use page.waitForTimeout() — it's a bad practice. Use web-first assertions instead.

Code Example

How do web-first assertions work and how are they different from standard assertions?IntermediateP1
Basic Answer

Web-first assertions (toBeVisible, toHaveText) keep trying until the condition is met or time runs out. Standard assertions (expect(value).toBe()) check once and fail immediately. Use expect.poll() or expect.toPass() for custom retry logic.

Code Example
// Web-first assertions — auto-retry up to 5 seconds (default timeout) await expect(page.getByRole('heading')).toBeVisible(); await expect(page.getByTestId('status')).toHaveText('Active'); await expect(page.getByRole('button')).toBeEnabled(); // Standard (non-web) assertion — checks once, no retry const url = page.url(); expect(url).toContain('/dashboard'); // Custom retry with expect.toPass (great for flaky operations) await expect(async () => { const response = await request.get('/api/status'); expect((await response.json()).state).toBe('COMPLETED'); }).toPass({ timeout: 10000 });

Explain Playwright fixtures — how they differ from beforeEach/afterEach.AdvancedP1
Basic Answer

Fixtures are a type-safe way to inject dependencies into tests. They combine setup AND cleanup in one place. They have different scopes (per-test or per-worker). Fixtures can depend on other fixtures. They're created only when needed (lazy). This removes the need for beforeEach/afterEach hooks.

Code Example
// BAD — beforeEach/afterEach pattern (manual, error-prone) let db; test.beforeEach(async () => { db = await Database.connect(); await db.seed(); }); test.afterEach(async () => { await db.cleanup(); // Easy to forget! }); test('user count', async () => { console.log(await db.countUsers()); }); // GOOD — Fixture pattern (setup + cleanup in one place, type-safe) const test = base.extend({ db: [async ({}, use) => { const db = await Database.connect(); await db.seed(); await use(db); // Give to test await db.cleanup(); // Auto-cleanup runs even if test fails! }, { scope: 'worker' }], // Created once per worker }); test('user count', async ({ db }) => { console.log(await db.countUsers()); });

How do you handle authentication across tests using storageState?AdvancedP1
Basic Answer

Login once in global setup, save cookies/localStorage to storageState.json. Set up in playwright.config.ts for each project. This cuts login time from ~3s to ~50ms per test. Use separate state files for different user roles (admin, user, etc.).

Code Example
// global-setup.ts — login once, save state async function globalSetup() { const browser = await chromium.launch(); const page = await browser.newPage(); await page.goto('https://app.com/login'); await page.getByLabel('Username').fill('admin'); await page.getByLabel('Password').fill('password'); await page.getByRole('button', { name: 'Sign In' }).click(); await page.waitForURL('**/dashboard'); // Save cookies + localStorage to a file (~50ms per test to reuse) await page.context().storageState({ path: 'state/admin-auth.json' }); await browser.close(); } // playwright.config.ts — reuse saved state projects: [ { name: 'admin', use: { storageState: 'state/admin-auth.json' } }, { name: 'user', use: { storageState: 'state/user-auth.json' } }, ] // Test file — no login code needed! test('dashboard loads', async ({ page }) => { await page.goto('/dashboard'); // Already authenticated await expect(page.getByRole('heading')).toBeVisible(); });

How does network interception work using page.route()?AdvancedP1
Basic Answer

page.route(pattern, handler) lets you intercept network requests. The handler can: fulfill (return mock data), continue (let it through, maybe modified), abort (block the request), or replay from a HAR file. This enables API mocking, modifying requests/responses, network throttling, and offline testing.

Code Example
// Mock an API response (fulfill) await page.route('**/api/users', async route => { await route.fulfill({ status: 200, body: JSON.stringify([{ id: 1, name: 'Mock User' }]), }); }); // Block analytics scripts (abort) await page.route('**/analytics.js', async route => route.abort()); // Modify request headers (continue with modifications) await page.route('**/api/**', async route => { const headers = { ...route.request().headers(), authorization: 'Bearer token' }; await route.continue({ headers }); }); // Simulate slow network (delay the response) await page.route('**/api/data', async route => { await new Promise(r => setTimeout(r, 5000)); // 5 second delay await route.continue(); });

Basic Answer

How does Playwright's fixture system enable dependency injection? Design a multi-layer fixture chain for enterprise testing.ExpertP1
Basic Answer

Fixtures in Playwright are a dependency injection system. Each fixture can depend on other fixtures, and Playwright resolves the dependency graph automatically. A multi-layer chain: config → auth → db → apiClient → page → authenticatedPage. Fixtures have scope: 'test' (per test) or scope: 'worker' (per worker). The use() callback provides setup AND teardown in one construct.

Detailed Answer
import { test as base } from '@playwright/test'; // Layer 1: Config fixture (worker-scoped, loaded once) const test = base.extend<{ config: TestConfig; db: Database; api: ApiClient; authPage: Page }>({ config: [async ({}, use) => { const config = await loadConfig(); await use(config); }, { scope: 'worker' }], // Layer 2: DB fixture depends on config db: [async ({ config }, use) => { const db = await Database.connect(config.dbUrl); await db.seedTestData(); await use(db); await db.cleanup(); // Auto-teardown }, { scope: 'worker' }], // Layer 3: API client depends on config api: [async ({ config }, use) => { const api = new ApiClient(config.baseUrl); await api.login(config.adminUser); await use(api); }, { scope: 'worker' }], // Layer 4: Authenticated page — depends on page + api authPage: [async ({ page, api }, use) => { const token = await api.getToken(); await page.goto('/login'); await page.evaluate(`localStorage.setItem('token', '${token}')`); await page.goto('/dashboard'); await use(page); }, { scope: 'test' }] }); // Test uses only the fixtures it needs test('user dashboard loads', async ({ authPage, db }) => { await expect(authPage.locator('h1')).toHaveText('Dashboard'); });
⚠️ Optimization Worker-scoped fixtures reduce setup overhead. A DB connection per worker (~5-10s) vs per test (~5-10ms context creation). Use scope: 'worker' for expensive setup, scope: 'test' for test-specific state.
Real Interview Answer

"I design fixture chains based on scope and dependency. Worker-scoped fixtures handle expensive setup like DB connections, API tokens, and config loading — once per worker. Test-scoped fixtures handle per-test state like authenticated pages with unique user credentials. The key insight is that fixtures auto-resolve dependencies, provide built-in teardown, and eliminate the need for beforeEach/afterEach hooks. In an enterprise framework with 30+ microservices, this pattern reduced our test setup time by 60%."

Design a custom Playwright reporter. How would you stream real-time results to a custom dashboard?ExpertP1
Basic Answer

Implement the Reporter interface from @playwright/test/reporter. Key methods: onBegin(), onTestBegin(), onTestEnd(), onEnd(), onStepBegin(), onStepEnd(), onError(). Each receives detailed test result objects with status, duration, errors, attachments, and annotations. Stream results via WebSocket, HTTP POST, or write to a message queue.

Detailed Answer
import { Reporter, TestCase, TestResult, FullConfig, Suite } from '@playwright/test/reporter'; class CustomDashboardReporter implements Reporter { private apiUrl: string; private results: TestResult[] = []; onBegin(config: FullConfig, suite: Suite) { this.apiUrl = process.env.DASHBOARD_URL || 'http://localhost:3000/api/results'; console.log(`Starting suite: ${suite.title}`); } onTestEnd(test: TestCase, result: TestResult) { const payload = { title: test.title, status: result.status, duration: result.duration, errors: result.errors.map(e => e.message), retry: result.retry, project: test.parent.project()?.name, tags: test.tags, timestamp: new Date().toISOString() }; // Non-blocking fire-and-forget POST fetch(this.apiUrl, { method: 'POST', body: JSON.stringify(payload), headers: { 'Content-Type': 'application/json' } }).catch(() => {}); // Don't let reporting failure affect tests } onEnd() { const summary = { total: this.results.length, passed: this.results.filter(r => r.status === 'passed').length, failed: this.results.filter(r => r.status === 'failed').length, duration: this.results.reduce((sum, r) => sum + r.duration, 0) }; console.log(`Suite complete: ${JSON.stringify(summary)}`); } } export default CustomDashboardReporter;

Config in playwright.config.ts: reporter: [['html'], ['./CustomDashboardReporter.ts']]. Multiple reporters run simultaneously.

Auth Strategy

Global setup → storageState.json → projects with storageState → no login per test

Fixture Pattern

test.extend({ fixture: async ({dep}, use) => { setup; use(val); cleanup; } })

Workers (file-level), fullyParallel (within-file), Sharding (multi-machine)
