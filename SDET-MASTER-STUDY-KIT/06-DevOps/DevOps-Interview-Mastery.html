P1 — Frequently Asked Questions

Must-know DevOps concepts for SDET interviews — these appear in every technical round.

1
Explain Git workflow strategies — GitFlow, GitHub Flow, and Trunk-Based Development.IntermediateP1
Basic Answer

GitFlow: Multiple branch types (main, develop, feature, hotfix, release). Organized but complex. Best for versioned releases in enterprise. GitHub Flow: Feature branches → main. Simple, best for continuous deployment (SaaS). Trunk-Based Development: Short-lived branches or commits directly to main with feature flags. Fastest, best for small teams with CI/CD. For SDET teams, GitHub Flow is most common. Pick a strategy that matches how often you deploy — the more frequent the deploys, the simpler the strategy.

2
What is the difference between git merge and git rebase? When would you use each?IntermediateP1
Basic Answer

git merge creates a merge commit that keeps all branch history. git rebase rewrites history by replaying commits on top of another branch — making a clean, straight line of commits. Use merge for shared branches (preserves history). Use rebase for local branches before pushing (cleaner history). Golden rule: never rebase shared branches. Rebase conflicts are fixed per commit, not all at once. In tests: rebase your feature branch before creating a PR to keep history clean.

3
What is CI vs CD vs Continuous Deployment? How does each apply to test automation?IntermediateP1
Basic Answer

CI (Continuous Integration): Build and test every commit automatically. For SDET: run unit tests, lint, type-check on every PR. CD (Continuous Delivery): CI + automatically deploy to staging, manual approval for production. SDET: E2E tests run after staging deploy. CD (Continuous Deployment): Fully automated to production. SDET: canary analysis, synthetic monitoring. Most companies use CI + Delivery. SaaS companies use Continuous Deployment. Key for SDET: tests must be fast enough (or parallel enough) to not slow down the pipeline.

4
Design a CI/CD pipeline for a Playwright framework with build, test, package, and deploy stages.AdvancedP1
Basic Answer

Stages: (1) Checkout — clone the repo. (2) Install — npm ci with caching (node_modules + Playwright browsers). (3) Lint — ESLint, Prettier, TypeScript check. (4) Build — compile TypeScript. (5) Test — npx playwright test --shard=X/4 across 4 parallel jobs. (6) Report — publish HTML report. (7) Package — docker build -t tests. (8) Deploy — push Docker image to registry. Use matrix builds for cross-browser. Retry on failure (max 2). Cache: node_modules, Playwright browsers.

5
How do you write a Jenkins Declarative Pipeline? Include checkout, build, test, archive, and notification.AdvancedP1
Basic Answer
pipeline { agent { docker { image 'mcr.microsoft.com/playwright:v1.40.0-focal' } } environment { CI = 'true' } stages { stage('Checkout') { steps { checkout scm } } stage('Install') { steps { sh 'npm ci' } } stage('Test') { parallel { stage('Shard 1') { steps { sh 'npx playwright test --shard=1/4' } } stage('Shard 2') { steps { sh 'npx playwright test --shard=2/4' } } stage('Shard 3') { steps { sh 'npx playwright test --shard=3/4' } } stage('Shard 4') { steps { sh 'npx playwright test --shard=4/4' } } } } stage('Report') { steps { publishHTML([reportDir: 'playwright-report', reportName: 'Report']) } } } post { failure { slackSend(color: 'danger', message: "Tests failed: ${env.BUILD_URL}") } success { slackSend(color: 'good', message: "Tests passed: ${env.BUILD_URL}") } } }
Detailed Answer

Declarative vs Scripted: Declarative enforces a rigid structure but is easier to maintain and read. Scripted offers maximum Groovy flexibility but is harder to debug. For SDET pipelines, Declarative is preferred for readability. Key features: post blocks for cleanup regardless of outcome, parallel for test sharding, docker agent for consistent environments. Best practices: use when conditions for branch-specific stages, input for manual approval gates, tools for Node.js version management.

6
How do you Dockerize a Playwright test framework? Write a multi-stage Dockerfile.AdvancedP1
Basic Answer
# Stage 1: Install dependencies FROM mcr.microsoft.com/playwright:v1.40.0-focal AS deps WORKDIR /app COPY package*.json ./ RUN npm ci # Stage 2: Run tests FROM deps AS runner COPY . . RUN npx playwright test
Detailed Answer

Multi-stage build benefits: smaller final image (only runtime deps), faster rebuilds (layer caching), security (no build tools in final image). Best practices: (1) Use .dockerignore to exclude node_modules, test reports, git history. (2) Use specific image tags (not :latest) for reproducibility. (3) Separate npm ci from COPY . to cache deps. (4) Set CI=true environment variable. (5) For CI/CD, mount volumes for reports: docker run -v $(pwd)/report:/app/playwright-report tests.

7
How does Docker differ from a virtual machine? Explain Docker architecture.IntermediateP1
Basic Answer

Docker containers share the host computer's operating system kernel — they're just isolated processes. Virtual Machines include a full guest OS with extra overhead. Docker architecture: Client (CLI/API) → Docker Daemon (builds/runs containers) → Registry (stores images) → Containers (isolated processes). Images are built in layers — each command creates a layer. Layers are cached for faster builds. For Playwright: Docker ensures the same browser environment on every machine and in CI.

8
What is Kubernetes? Explain the architecture — control plane, worker nodes, pods, deployments, services.AdvancedP1
Basic Answer

Control Plane: API Server (all communication), Scheduler (assigns work), Controller Manager (maintains state), etcd (key-value store for cluster state). Worker Nodes: Kubelet (node agent), Kube Proxy (networking), Container Runtime (Docker/containerd). Pod: smallest deployable unit (one or more containers, shared network). Deployment: manages ReplicaSets for rolling updates and rollbacks. Service: stable network endpoint for pods (ClusterIP, NodePort, LoadBalancer). For test automation: use K8s Jobs for batch test runs, Deployments for long-running services (test dashboards, mock servers).

9
How do you run Playwright tests in Kubernetes? Design a K8s test execution grid.ExpertP1
Basic Answer

Create Docker image with Playwright + tests. Use Kubernetes Job for execution. ConfigMap for env vars. Secrets for credentials. HPA for auto-scaling based on queue depth. Example: K8s Job with parallelism: spec.parallelism: 10 splits tests across pods using --shard. PersistentVolume for report storage. Ingress for web-based report access. Helm chart for reproducible deployment. Key considerations: resource limits per pod (CPU/memory), node affinity for specific browsers, pod anti-affinity to avoid running multiple test pods on the same node (for resource contention).

10
How do you handle flaky tests in CI/CD? Implement retry strategies with backoff.AdvancedP1
Basic Answer

Multiple levels: (1) Test-level: Playwright retries (retries: 2 in config). (2) Operation-level: expect.toPass() for flaky assertions. (3) Pipeline-level: re-run failed tests on a separate agent. (4) Auto-quarantine: if test fails >3 times in a row, auto-quarantine and alert. Backoff: first retry immediately, second after 5s, third after 30s (exponential + jitter). Track flakiness metrics per test (passes on retry / total executions). Establish a flakiness budget (e.g., <5%). Automatically quarantine tests exceeding the budget and notify team.

P2 — Commonly Asked Questions

11
Explain git stash, git cherry-pick, git revert, and git reset with use cases.IntermediateP2
Basic Answer

git stash: temporarily save uncommitted changes (apply later). Use when you need to switch branches quickly. git cherry-pick: apply specific commit(s) from another branch. Useful for backporting hotfixes. git revert: create a new commit that undoes a previous commit — safe for shared branches. git reset: move branch pointer backwards — modifies history (--soft: keep changes staged, --mixed: keep changes unstaged, --hard: discard changes). Use revert for public history, reset for local/private history only.

12
What are Kubernetes Services, Deployments, ReplicaSets, and Ingress? How do they relate?AdvancedP2
Basic Answer

Deployment manages ReplicaSets with desired pod count. ReplicaSet ensures n pod replicas running. Service provides stable networking to pods (ClusterIP internal, NodePort external, LoadBalancer for cloud). Ingress routes external HTTP/HTTPS to Services with host/path rules. Flow: Ingress → Service → Pod (managed by Deployment/ReplicaSet). For test automation: Deployments for test dashboard, Services for internal test tooling, Ingress for external report access.

13
How do you set up GitHub Actions or GitLab CI for Playwright test execution with sharding?IntermediateP2
Basic Answer

GitHub Actions: use matrix strategy: matrix: { shard: [1/4, 2/4, 3/4, 4/4] }. Use actions/cache for Playwright browsers. Upload reports as artifacts. GitLab CI: use parallel:matrix for sharding. Use the official Playwright Docker image. Best practice: run unit tests first (fast feedback), then Playwright (parallel). Cache Playwright browsers between runs to save ~15s per job. Store failed test traces as artifacts for debugging.

14
What is Docker Compose? How do you use it for local test environment orchestration?IntermediateP2
Basic Answer

Docker Compose defines multi-container applications in a docker-compose.yml file. For test automation: define app + database + mock server + test runner as separate services. depends_on ensures service startup order. healthcheck ensures services are ready before tests run. Example: docker-compose up -d --wait starts the full stack, then run tests. Benefits: consistent local environment, fully isolated, easy cleanup (docker-compose down -v).

15
What are blue-green and canary deployment strategies? How do you test them?AdvancedP2
Basic Answer

Blue-Green: two identical production environments. Switch traffic from Blue to Green. Instant rollback by switching back. Canary: gradually shift small % of traffic (e.g., 5%) to new version. Monitor metrics; if healthy, increase traffic. Testing: for blue-green, run smoke tests on green before switching traffic. For canary, validate that canary traffic produces correct results. Use traffic mirroring to test new version without impacting users. K8s: use Service Mesh (Istio) for fine-grained traffic splitting.

P3 — Valuable Context

16
What is GitOps? How does ArgoCD work with Kubernetes?ExpertP3
Basic Answer

GitOps uses Git as the single source of truth for declarative infrastructure. ArgoCD is a Kubernetes controller that continuously reconciles cluster state with Git repository state. Benefits: audit trail (every change is a commit), easy rollback (revert a Git commit), self-healing (auto-corrects drift). For SDET: GitOps ensures test infrastructure (test environments, mock servers, test dashboards) is version-controlled and auditable. Test config changes follow the same review process as application code.

17
How do you implement canary deployments with traffic splitting for testing?ExpertP3
Basic Answer

Canary deploys route a small % of traffic (e.g., 5%) to new version while keeping old version running. K8s: use Service Mesh (Istio/Linkerd) for fine-grained traffic splitting. Alternately: two Deployments with different labels, Service routes based on weights. For SDET: run automated smoke tests against canary before increasing traffic. Monitor error rate, latency, business metrics. Auto-rollback if thresholds breached. Gradual increase: 5% → 25% → 50% → 100% with health checks at each stage. Tools: Flagger (automated canary for K8s), Argo Rollouts.

18
What is Infrastructure as Code (IaC)? How does Terraform compare to CloudFormation?AdvancedP3
Basic Answer

IaC manages infrastructure (servers, networks, databases) through machine-readable definition files, version-controlled in Git. Terraform: cloud-agnostic (AWS, Azure, GCP), declarative HCL language, state management, plan/apply workflow. CloudFormation: AWS-only, YAML/JSON, native integration. For SDET: use Terraform to provision test environments (EC2, RDS, EKS clusters) on demand. Destroy after test completion to save costs. Terraform modules standardize test infrastructure across teams.
