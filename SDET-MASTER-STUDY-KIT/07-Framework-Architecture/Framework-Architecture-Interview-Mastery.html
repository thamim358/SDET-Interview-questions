P1 — Frequently Asked Questions

Framework design is the most critical skill for senior SDET roles — these questions differentiate junior from senior engineers.

1
What are the essential layers of a test automation framework? Explain separation of concerns.AdvancedP1
Basic Answer

Essential layers: (1) Test Layer — test specs with assertions. (2) Page/Component Object Layer — interaction methods. (3) Service Layer — API clients, DB helpers. (4) Fixture Layer — dependency injection. (5) Utility Layer — helpers, data generators, custom matchers. (6) Configuration Layer — environment configs, secrets. (7) Reporting Layer — loggers, reporters, screenshots. Each layer has a single responsibility and depends on the layer below it. This enables parallel development, easy debugging, and framework evolution.

2
Explain the Page Object Model (POM). What are its benefits, limitations, and alternatives?IntermediateP1
Basic Answer

POM encapsulates page structure and interactions in classes. Benefits: reduces duplication, improves maintainability, creates domain-specific language. Limitations: can become bloated for complex pages, encourages checking state after navigation (violating action verification). Alternatives: (1) Component-based — smaller focused objects for reusable components. (2) Screenplay Pattern — Actors perform Tasks using Abilities (more granular, better for complex workflows). (3) Flow Object — methods that return the next page object (fluent API). Choose based on application complexity and team size.

Explain design patterns used in test automation: Factory, Builder, Strategy, Singleton, Facade.ExpertP1
Basic Answer

Factory: creates page objects or API clients dynamically. Playwright fixtures are a factory pattern. Builder: constructs complex test data: new UserBuilder().withRole('admin').withEmail('test@test.com').build(). Strategy: interchangeable algorithms (different comparison strategies, locator strategies). Singleton: single instance of config manager or DB connection — use cautiously (test pollution risk). Facade: simplified interface to complex subsystems — service objects that combine multiple page operations (e.g., checkoutService.completePurchase()). Dependency Injection: Playwright fixtures are the best example — inject pages, API clients, configs into tests.

// Builder Pattern for test data class UserBuilder { private user = { role: 'user', active: true }; withRole(role: string) { this.user.role = role; return this; } withEmail(email: string) { this.user.email = email; return this; } build() { return { ...this.user }; } } // Strategy Pattern for comparison const comparisonStrategy = { exact: (a, b) => a === b, contains: (a, b) => a.includes(b), regex: (a, b) => new RegExp(b).test(a), };
4
How do you design a Hybrid Framework combining POM, data-driven, and keyword-driven approaches?AdvancedP1
Basic Answer

A hybrid framework combines the best of each: POM for UI interaction structure, data-driven via test.each() for parameterized execution from JSON/CSV/DB, keyword-driven via descriptive test names that read like action keywords, and fixture-driven via Playwright's dependency injection. Example: test reads data from a CSV → passes to a fixture → fixture uses page object methods → test asserts results. The framework adapts to the test type — simple UI test uses POM, API test uses service layer, data validation test uses DB layer.

5
How do you manage test data in a framework? Compare JSON, CSV, Faker, DB, and API approaches.AdvancedP1
Basic Answer

Each approach has trade-offs: JSON/CSV — simple, version-controlled, good for static data. Faker — generates random realistic data, great for negative/edge case testing. Database — single source of truth, requires DB setup in CI. API-based — create real data via APIs, most realistic. Recommendation: API-based setup in fixtures for critical paths (realistic), Faker for edge cases (randomized), static JSON for baseline configs. Avoid Excel in CI pipelines. Use the Builder pattern for composable test data with sensible defaults and overrides.

6
How do you design parallel execution with thread/browser isolation and data isolation?ExpertP1
Basic Answer

Three isolation levels: (1) Browser isolation — each worker gets a separate browser process with its own BrowserContext. (2) Data isolation — each test generates unique data (UUIDs, timestamps) or uses transaction rollback. (3) State isolation — storage state, cookies, localStorage are unique per context. Resource management: limit workers to CPU cores - 1, use Docker for resource limits, monitor memory. For 500+ parallel tests: use K8s pods with resource limits, each pod runs a subset of tests. ThreadLocal in Java or fixture scoping in Playwright handle thread-safe state.

7
Design a configuration management system for multiple environments (dev, staging, prod).AdvancedP1
Basic Answer

Layered configuration: (1) Base config — defaults that apply everywhere. (2) Environment config — overrides per environment. (3) Local config — developer overrides (gitignored). (4) Runtime config — environment variables (CI/CD secrets). TypeScript ensures type safety across all layers. config.ts reads process.env, merges with config files, validates at startup. Use as const and satisfies for compile-time validation. Example structure: config/ folder with base.ts, dev.ts, staging.ts, prod.ts.

8
Design a logging and reporting framework for test automation with CI integration.AdvancedP1
Basic Answer

Multi-level reporting: (1) Real-time — console output during execution (structured JSON logs). (2) HTML — Playwright's built-in HTML reporter with test filtering and trace viewer. (3) Dashboard — aggregated results across runs (Allure + Grafana). (4) CI artifacts — screenshots (only-on-failure), videos (on-first-retry), traces (on-first-retry). (5) Notifications — Slack/Teams with summary, trend graph, and failing test links. Store reports on S3/GCS with versioned directories per CI run. Use structured logging: { level, timestamp, testId, step, duration, error } for machine-parseable logs.

9
How do you implement exception handling and recovery in a test framework?AdvancedP1
Basic Answer

Categorize exceptions: Retryable (network timeout, stale element) — auto-retry with backoff. Fatal (auth failure, page not found) — fail immediately with clear message. Environment (server down, DB connection lost) — skip and report as infrastructure failure. Recovery: (1) Take screenshot on failure. (2) Log current URL and DOM state. (3) Restore browser context to clean state. (4) Continue with next test. Use Playwright's test.fail() for expected failures, test.fixme() for known issues with tickets. Never catch and swallow exceptions silently.

10
How do you implement dependency injection in a Playwright test framework using fixtures?AdvancedP1
Basic Answer

Playwright fixtures implement the DI pattern natively. test.extend() creates a new test instance with extended fixtures. Fixtures can depend on other fixtures (Playwright resolves the dependency graph). Scope: worker (shared across all tests in a worker) or test (created per test). Auto fixtures run automatically without being explicitly used. Example: authPage fixture depends on page fixture, handles login, and provides authenticated page to tests. This eliminates global beforeEach hooks and makes dependencies explicit.

import { test as base } from '@playwright/test'; type MyFixtures = { authedPage: Page; api: ApiClient; user: TestUser; }; const test = base.extend<MyFixtures>({ authedPage: async ({ page, user }, use) => { await page.goto('/login'); await page.getByLabel('Email').fill(user.email); await page.getByLabel('Password').fill(user.password); await page.getByRole('button', { name: 'Login' }).click(); await use(page); // test receives the authed page }, user: async ({ }, use) => { await use({ email: 'test@test.com', password: 'pass123' }); } });

P2 — Commonly Asked Questions

11
What is the Screenplay pattern? How does it compare to POM?ExpertP2
Basic Answer

Screenplay (Serenity/JS) is based on the concept of Actors who perform Tasks using Abilities to interact with Elements. Actors have Abilities (BrowseTheWeb, CallAnApi). Tasks are high-level actions (Login, SearchForProduct). Activities are low-level interactions (Click, Enter). Compared to POM: Screenplay is more granular, better for complex multi-step workflows, and avoids page object bloat. However, it has a steeper learning curve and more boilerplate. Use Screenplay for complex enterprise applications with many user roles and workflows.

12
How do you design a data-driven test framework with external data sources?AdvancedP2
Basic Answer

Use test.each() for inline parameterization. For external data: load from JSON/CSV files and generate test cases dynamically. Data sources: CSV (for tabular data, easy to share), JSON (for complex nested data), Excel (for business stakeholders — use caution in CI), Database (for data consistency). Challenges: test naming (each data set needs a unique, readable name), reporting (individual results per data set), parallelism (data-driven tests run in parallel within a file), failure isolation (one failure shouldn't stop others). Use Playwright's test.describe.configure({ mode: 'parallel' }).

13
How do you implement custom reporters and listeners in Playwright?AdvancedP2
Basic Answer

Implement Playwright's Reporter interface: onBegin, onTestBegin, onTestEnd, onEnd. Use for: (1) Custom notifications (Slack/Teams/email). (2) Database logging of test results. (3) Real-time dashboard updates. (4) Custom metrics calculation (flakiness score, execution velocity). Config: reporter: [['list'], ['custom-reporter.ts']]. Framework examples: Allure reporter, JUnit XML reporter for CI integration. Best practice: create a composite reporter that delegates to multiple reporters.

14
How do you create reusable utility classes without creating the "god class" anti-pattern?IntermediateP2
Basic Answer

Avoid a single Helper class with 50 unrelated methods. Instead: (1) Group by domain — stringUtils, dateUtils, fileUtils, apiUtils, assertionUtils. (2) Pure functions — no side effects, no state. (3) Composition over inheritance. (4) Prefer exports over classes — export individual functions. (5) Document each utility with examples, expected inputs, and outputs. Example: export function formatDate(date: Date, format: 'ISO' | 'display'): string rather than class Helper { formatDate()... }.

15
How do you design folder structure for a scalable test automation framework?IntermediateP2
Basic Answer

Standard structure: src/ — tests/ (spec files), pages/ (page/component objects), api/ (API client + service objects), fixtures/ (custom fixtures), data/ (test data JSON/CSV), utils/ (helpers, generators, matchers), config/ (environment configs), reporters/ (custom reporters). Root: playwright.config.ts, package.json, Dockerfile, tsconfig.json, .env (gitignored). For multi-product: products/productA/, products/productB/. For microservices: services/serviceA-api/. This structure scales to 100k+ tests across multiple products.

P3 — Valuable Context

16
How do you implement self-healing locators with attribute-based fallback strategies?ExpertP3
Basic Answer

Create a wrapper around Playwright locators that implements a fallback chain: try getByTestId → getByRole → getByText → locator(css). If all fail, use ML-based heuristics (find closest element matching original attributes by training on successful locators). Log each fallback as a warning — track patterns to recommend locator updates. Playwright's built-in auto-healing via Trace Viewer helps identify what changed. Implement a locator health score: track time-to-find, fallback count, and success rate for each locator.

17
How do you implement flaky test detection and auto-quarantine in your framework?AdvancedP3
Basic Answer

Track flakiness per test: passes on retry vs first attempt over the last 10 runs. Auto-quarantine: if test passes on retry >2 times in 10 runs, add to quarantine list (skip in CI, continue running in background). Send alert to team. After fixing, manually remove from quarantine. Use a CI step that checks a flakiness database and updates Playwright's test.skip() annotations automatically. Flakiness score: (passes on retry / total runs) × 100. Budget: <5% overall flakiness. Tests above 30% auto-quarantine.

18
How do you implement API mocking and service virtualization in your test framework?AdvancedP3
Basic Answer

Playwright's page.route() for client-side mocking. For server-side/service virtualization: (1) WireMock — standalone HTTP mock server with stub definitions. (2) MSW (Mock Service Worker) — intercepts at the network level in the browser. (3) Testcontainers — spin up real service containers for integration tests. (4) Contract testing (Pact) — ensures mocks stay in sync with real services. Strategy: use Playwright route() for UI tests, WireMock for API integration tests, Testcontainers for E2E with real dependencies.
