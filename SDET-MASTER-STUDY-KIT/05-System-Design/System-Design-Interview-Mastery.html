P1 — Frequently Asked System Design Questions

Design problems that appear in senior SDET and Test Architect interviews at top tech companies.

Design a scalable Playwright automation framework from scratch for a microservices-based e-commerce application.ExpertP1
Basic Answer

Key design points: Separate folders for page objects, components, API services, utilities, fixtures, and tests. Use Playwright fixtures for dependency injection. Type-safe configs for different environments. Auth via storageState. Component library for shared UI elements. Service objects for API calls. Data factories for test data. Parallel execution via workers. CI/CD with sharding and retries. Architecture layers: Tests → Fixtures → Page Objects + API Services → Playwright → Browser Contexts → CI/CD.

2
Design a test execution platform that runs 10,000+ tests daily across multiple browsers on Kubernetes.ExpertP1
Basic Answer

Main parts: (1) Test scheduler — sends tests to worker pods. (2) Worker pool — Kubernetes pods running Playwright in Docker. (3) Queue — manages test jobs with priorities (RabbitMQ/Redis). (4) Result aggregator — collects results. (5) Retry manager — retries flaky tests with increasing delays. (6) Dashboard — real-time view (Grafana). Uses K8s with auto-scaling based on queue size. Tests split using --shard. Each pod runs isolated browser contexts. Goal: 10,000 tests in ~20 min.

3
Design a CI/CD pipeline with quality gates that balances deployment velocity with safety.AdvancedP1
Basic Answer

Pipeline stages: (1) Commit — lint, type-check, unit tests (under 5 min). Must pass. (2) Integration — API tests, contract tests (under 15 min). Must pass. (3) E2E — Playwright tests on critical paths, sharded (under 20 min). Must pass. (4) Staging deploy — push to staging. (5) Regression — full regression suite, runs async (under 60 min). Reports but doesn't block. (6) Prod deploy — blue-green with canary analysis. Quality gates: no critical bugs, flakiness under 5%, API coverage over 80%, no performance drop. Run fast tests early for quick feedback.

4
Design a test data management strategy for 500+ parallel tests requiring complete data isolation.AdvancedP1
Basic Answer

Strategy layers: (1) Unique data — each test generates data with UUIDs/timestamps to avoid conflicts. (2) DB transactions — wrap tests in transactions that roll back (fastest, but doesn't catch all issues). (3) API setup — create data via APIs in fixtures (most realistic). (4) Docker databases — each K8s pod gets its own test DB container. (5) Seeding — baseline data per test suite. Cleanup via TRUNCATE or transaction rollback. For 500 parallel tests: use connection pools and avoid shared state. PII: mask/anonymize production data in test environments.

5
Design a distributed cross-browser and cross-device test execution grid.ExpertP1
Basic Answer

Architecture: (1) Router — sends test requests to available nodes. (2) Node Manager — manages Docker worker nodes with pre-warmed browsers. (3) Browser Pool — Chromium, Firefox, WebKit contexts at different screen sizes, ready to go instantly. (4) Device Profiles — mobile viewports, touch, geolocation. (5) Results Collector — central storage. Use Docker Swarm or K8s. Scale up/down based on queue size. For cloud: integrate with BrowserStack as backup.

6
Design a multi-tenant automation framework supporting multiple products and environments.ExpertP1
Basic Answer

Core design: (1) Config hierarchy — default → environment → product → test overrides. (2) Product abstraction — shared base with product-specific page objects/fixtures. (3) Environment management — typed config per environment (dev/staging/prod). (4) Feature flags — toggle product-specific features. (5) Reporting — separate dashboards per product. (6) Test categorization — tags for product, environment, priority. Key rule: the core framework should never change for product-specific tests.

7
Design a cloud-native QA platform using Docker and Kubernetes for automated test execution.ExpertP1
Basic Answer

Components: (1) Docker images — Playwright tests packaged as Docker image (multi-stage). (2) K8s Jobs/CronJobs — for running tests. (3) ConfigMaps/Secrets — environment configs and credentials. (4) PVC — storage for reports and artifacts. (5) Ingress/Service — for reporting dashboard. (6) HPA — auto-scale based on queue size. Flow: CI builds Docker image → pushes to registry → K8s creates Job → Pods run tests in parallel → results saved → notification sent. Use Helm charts for repeatable deployments.

8
Design a reporting and analytics platform for automation results across multiple teams.AdvancedP1
Basic Answer

Key features: (1) Result ingestion — real-time streaming from CI/CD. (2) Storage — time-series DB for trends (InfluxDB), relational DB for metadata (Postgres). (3) Dashboards — pass/fail rates, flakiness trends, execution times, coverage. (4) Analytics — flaky test detection, regression alerts, performance changes. (5) Team views — per-team dashboards with week-over-week comparison. Use Elasticsearch for search, Grafana for dashboards. Alert on: >5% failure rate increase, >20% execution time increase.

9
Design a visual testing platform with baseline comparison and diff management.AdvancedP1
Basic Answer

Components: (1) Baseline storage — S3/GCS for screenshots, versioned per app version. (2) Comparison engine — pixel diff with adjustable threshold. (3) Diff management — review UI for approving/rejecting changes, with before/after views. (4) CI integration — fail on diff, allow manual approval. (5) Automation — Playwright's toHaveScreenshot. Strategy: capture at multiple screen sizes, disable animations, use consistent test data. Store baseline per browser + viewport + component.

10
Design a self-healing automation framework that handles flaky tests and dynamic locators.ExpertP1
Basic Answer

Self-healing strategies: (1) Locator hierarchy — getByRole → getByTestId → getByText → locator(fallback). (2) Attribute fallback — if primary fails, try secondary attributes (title, aria-label, class). (3) ML-based heuristics — train on successful locators to predict alternatives. (4) Auto-quarantine — if test fails >2 times consecutively, auto-quarantine and alert. (5) Historical analysis — track locator stability, suggest replacements for frequently failing locators. Implementation: wrapper around Playwright locator that catches TimeoutError and tries alternatives before failing.

P2 — Commonly Asked Questions

11
Design an observability and monitoring strategy for test execution in production.AdvancedP2
Basic Answer

Three pillars: (1) Logging — structured JSON logs per test step, stored in Elasticsearch. (2) Metrics — execution time, pass/fail rates, flakiness index (passes on retry / total runs), test coverage %. (3) Tracing — distributed tracing across API calls with Playwright Trace Viewer. Production monitoring: synthetic monitoring running health check scripts every 5 minutes from multiple cloud regions. Alert on: >2% failure rate increase, >10s p95 response time, critical user journey failures. Use OpenTelemetry for cross-service tracing.

12
Design a contract testing strategy for microservices using Pact.ExpertP2
Basic Answer

Consumer-Driven Contracts: Provider publishes contract, consumer verifies against it. Pact workflow: Provider tests generate contracts (can I deploy?) → uploaded to Pact Broker → Consumer tests verify against contracts (can I deploy?). Integration: Playwright API tests use generated contracts to validate responses. CI: verify contracts in pipeline before deployment — if contract breaks, the breaking change can't be deployed. Benefits: catch breaking API changes early, reduce integration testing effort, enable parallel development. SDET role: maintain contract test suite, manage Pact Broker.

13
Design a synthetic monitoring platform using Playwright for production health checks.AdvancedP2
Basic Answer

Architecture: (1) Scheduler — runs health checks every 5 minutes (K8s CronJob). (2) Playwright scripts — critical user journeys: login, search, add to cart, checkout. (3) Multi-region — run from 3+ geographic locations (us-east, eu-west, ap-southeast). (4) Alerting — PagerDuty/Slack on failure with trace attachment. (5) Dashboard — uptime %, response time trends (p50/p95/p99), error breakdown by region. Store results in time-series DB. Compare performance against baseline and week-over-week trends.

14
Design a mobile device farm for Android and iOS automation (real devices + cloud).AdvancedP2
Basic Answer

Farm architecture: (1) Device pool — real devices for critical paths, emulators/simulators for parallel execution. (2) Connection manager — ADB for Android, WebDriverAgent for iOS. (3) Session allocation — queue-based, one session per device. (4) Device provisioning — automated OS update, app install, reset between tests. (5) Cloud integration — BrowserStack/Sauce Labs as overflow. Key metrics: device utilization %, session wait time, test execution time. Containerize Appium servers per device using Docker.

15
Design a test strategy for event-driven architectures (Kafka, SQS, event streams).ExpertP2
Basic Answer

Testing layers: (1) Producer validation — verify messages published with correct schema (Avro/Protobuf) and content. (2) Consumer validation — verify messages consumed and processed correctly. (3) Schema compatibility — backward/forward compatibility tests for schema registry. (4) Ordering — test message ordering within partitions. (5) Error handling — poison pill messages, retry logic, dead letter queue. (6) Performance — throughput under load, latency percentiles. Use testcontainers for local Kafka/SQS emulation in CI. Assert on event counts and content.

P3 — Valuable Context

16
Design a chaos engineering testing strategy for a distributed microservices system.ExpertP3
Basic Answer

Chaos testing validates system resilience. Experiments: (1) Service failures — kill random service instances (via K8s). (2) Network latency — inject 500ms-2s delay on inter-service calls. (3) Resource exhaustion — limit CPU/memory/disk on pods. (4) Database failures — simulate DB connection drops, slow queries. (5) Region failures — simulate an entire AWS availability zone outage. Tools: Chaos Mesh (K8s), Litmus, Gremlin. Run experiments in staging first, then in production with limited blast radius. Key metric: MTTR (mean time to recover).

17
How would you design a performance testing strategy for a high-traffic API gateway?ExpertP3
Basic Answer

Approach: (1) Define SLOs — latency p95 < 200ms, throughput > 1000 req/s, error rate < 0.1%. (2) Load profiles — ramp-up test (find breaking point), spike test (sudden load increase), soak test (sustained load for 4h). (3) Tools — k6 for load generation, Playwright for synthetic UI checks under load. (4) Metrics — response time distribution, CPU/memory per pod, connection pool utilization, DB query performance. (5) CI integration — run performance tests nightly, compare against baseline. Alert on >5% regression.
