P1 — Frequently Asked Questions

Must-know for TypeScript-focused SDET interviews.

1
What are the advantages of TypeScript over JavaScript for large-scale test automation projects?IntermediateP1
Basic Answer

TypeScript provides static typing, catching errors at compile time rather than runtime. This is critical for large test suites where refactoring without type safety risks silent failures. Benefits include better IDE support (autocompletion, refactoring), self-documenting code through type definitions, reduced runtime errors from type mismatches, and easier onboarding for new team members. For Playwright specifically, TypeScript enables type-safe locators, page objects, and API response validation.

Real Interview Answer

"TypeScript catches category-level bugs during compilation that would otherwise cause test failures at runtime. For example, if an API response shape changes, TypeScript surfaces the issue immediately across all tests using that response type. Combined with Playwright's type-safe locators, this drastically reduces flaky tests caused by type mismatches."

2
Explain the difference between interface and type alias. When would you use each?AdvancedP1
Basic Answer

Both define object shapes, but key differences exist. interface supports declaration merging (reopening to add properties) and extends more naturally with extends. type aliases can represent unions, intersections, primitives, and tuples — they are more flexible for composite types. Use interface for public API contracts and object shapes that might be extended. Use type for unions, intersections, complex types, or when you need computed properties.

Detailed Answer
// Interface — extensible, declaration merging interface User { id: number; name: string; } interface User { email: string; } // Merged — User now has id, name, email interface AdminUser extends User { role: 'admin'; } // Type — flexible, can use unions/intersections type Status = 'active' | 'inactive' | 'pending'; type ApiResponse<T> = { data: T; error: null } | { data: null; error: string }; type StringOrNumber = string | number; type UserWithRole = User & { role: string }; // Intersection
Real Interview Answer

"In my Playwright framework, I use interfaces for page object contracts — each page interface defines what methods the page exposes. I use types for API response shapes, discriminated unions for test outcomes, and template literal types for dynamic locator patterns. The rule of thumb: interface for objects that other code depends on, type for everything else."

3
What are generics? Implement a generic API response wrapper type.AdvancedP1
Basic Answer

Generics allow creating reusable components that work with multiple types while preserving type information. They are parameterized types defined with angle brackets <T>. A generic API response wrapper ensures type safety across all API calls in a test framework.

Detailed Answer
interface ApiResponse<T> { status: number; message: string; data: T; timestamp: string; } interface PaginatedResponse<T> extends ApiResponse<T[]> { pagination: { page: number; totalPages: number; totalItems: number; }; } // Usage in test framework interface User { id: number; name: string; email: string; } async function getUser<T>(id: number): Promise<ApiResponse<T>> { const response = await request.get(`/api/users/${id}`); return response.json(); } // Type-safe — TypeScript knows user.data has id, name, email const user = await getUser<User>(1); expect(user.data.name).toBe('Alice');
4
What are union types and intersection types? Give real-world examples from test automation.IntermediateP1
Basic Answer

Union types (A | B) allow a value to be one of several types. Intersection types (A & B) combine multiple types into one. In test automation, unions model different API response states (success/failure), and intersections combine test configuration with environment variables.

type TestResult = { passed: true; duration: number } | { passed: false; error: string }; type Config = BrowserConfig & EnvironmentConfig & AuthConfig; type LocatorStrategy = 'role' | 'testid' | 'text' | 'css' | 'xpath';
5
How does type narrowing work? Explain typeof, instanceof, in, and discriminated unions.AdvancedP1
Basic Answer

Type narrowing is TypeScript's process of refining a broader type to a more specific type within a code block. typeof x === 'string' narrows to string, instanceof Error narrows to Error class, 'property' in obj checks for property existence. Discriminated unions use a literal property (the discriminant) to narrow — most powerful pattern for API response handling.

type ApiState<T> = | { status: 'loading' } | { status: 'success'; data: T } | { status: 'error'; error: string }; function handleState<T>(state: ApiState<T>) { if (state.status === 'loading') return 'Loading...'; // Narrowed: { status: 'loading' } if (state.status === 'success') return state.data; // Narrowed: { status: 'success'; data: T } return state.error; // Narrowed: { status: 'error'; error: string } }
6
What are utility types? Explain Partial<T>, Required<T>, Pick<T>, Omit<T>, Record<K,T>, and Readonly<T>.IntermediateP1
Basic Answer

Utility types are built-in generic transformations. Partial<T> makes all properties optional — useful for test data overrides. Required<T> makes all required. Pick<T, K> selects specific keys. Omit<T, K> removes specific keys. Record<K,T> creates an object type with keys K and values T. Readonly<T> makes properties immutable.

interface User { id: number; name: string; email: string; role: string; } type PartialUser = Partial<User>; // All optional — for test data overrides type UserPreview = Pick<User, 'id' | 'name'>; // Just id and name type UserWithoutRole = Omit<User, 'role'>; // All except role type UserMap = Record<number, User>; // { [key: number]: User }
7
What is the difference between any and unknown? When would you use each?IntermediateP1
Basic Answer

any disables all type checking — you can access any property, call it as a function, or assign it to any type. unknown is the type-safe counterpart — you must narrow it before use (via typeof, type guards, or assertions). Never use any in production code. Use unknown for external data (API responses, user input) where the shape isn't guaranteed.

⚠️ Interview Trap Saying "I use any occasionally to save time" is a red flag. Always prefer unknown and narrow it. If you use any, justify it explicitly (e.g., "in a migration" or "for a third-party library with incomplete types").
8
What are mapped types? Create a custom mapped type.ExpertP1
Basic Answer

Mapped types iterate over property keys of a type to create new types. Syntax: { [P in K]: T }. The built-in Partial, Readonly, Pick, and Record are all mapped types. Custom mapped types can transform property modifiers or values.

type Nullable<T> = { [K in keyof T]: T[K] | null }; type Stringify<T> = { [K in keyof T]: string }; type DeepReadonly<T> = { readonly [K in keyof T]: T[K] extends object ? DeepReadonly<T[K]> : T[K]; };
9
What are conditional types? Explain with the infer keyword.ExpertP1
Basic Answer

Conditional types select types based on a condition: T extends U ? X : Y. The infer keyword declares a type variable within a conditional type's extends clause, allowing you to extract and use types from within other types. ReturnType<T> and Parameters<T> use infer internally.

type IsString<T> = T extends string ? 'yes' : 'no'; // Extract Promise value type type Unwrap<T> = T extends Promise<infer U> ? U : T; type Val = Unwrap<Promise<number>>; // number // Extract function return type type MyReturnType<T> = T extends (...args: any[]) => infer R ? R : never;
10
How do you type API responses in Playwright? Design a generic API response type.AdvancedP1
Basic Answer

Playwright's APIResponse provides generic methods. Create a typed wrapper that parses and validates the response shape at compile time. Use discriminated unions for success/error states and utility types for pagination.

interface ApiClient { get<T>(url: string): Promise<{ data: T; status: number }>; post<T, B>(url: string, body: B): Promise<{ data: T; status: number }>; } class PlaywrightApiClient implements ApiClient { constructor(private request: typeof request) {} async get<T>(url: string) { const response = await this.request.get(url); const data: T = await response.json(); return { data, status: response.status() }; } }

P2 — Commonly Asked Questions

Beginner to intermediate questions frequently appearing in technical screens.

11
What are the basic types in TypeScript? How do you annotate variables and function parameters?BeginnerP2
Basic Answer

Basic types include string, number, boolean, null, undefined, void, and any. Annotate variables with let name: string = 'Alice' and parameters with function greet(name: string): string { ... }. TypeScript can also infer types, making annotations optional in many cases. For Playwright tests, annotating function parameters ensures type-safe page object methods.

Real Interview Answer

"Basic type annotations catch bugs early. For example, if a function expects string but gets number, TypeScript flags it at compile time before the test even runs. In Playwright page objects, I annotate all method parameters and return types — this makes the framework self-documenting and prevents runtime type errors."

12
How does type inference work? When do you need explicit type annotations vs letting TS infer?BeginnerP2
Basic Answer

TypeScript infers types from values. let count = 5 infers number. Inference works well for simple variable initialization. Use explicit annotations when: (1) a function returns multiple possible types, (2) defining complex object shapes (interfaces), (3) function parameters and return types for public APIs, (4) when inference would be too wide (e.g., let data = {} infers as {} not a specific shape). In Playwright, inference works for simple locator assignments, but explicit types are better for fixture definitions and page objects.

// TypeScript infers: string let name = 'Alice'; // Explicit annotation needed for complex types interface User { id: number; name: string; } const getUser = async (): Promise<User> => { ... }; // Number is too broad — use literal type let port: 3000 | 3001 | 3002 = 3000;
13
What are arrays and tuples in TypeScript? How do they differ?BeginnerP2
Basic Answer

Arrays have variable length and uniform element types: string[] or Array<number>. Tuples have fixed length and known element types at each position: [string, number]. Tuples enforce both position and type — [3, 'hello'] would error if the type is [string, number]. In test automation, tuples are useful for parameterized test cases where each parameter has a different type.

type TestCase = [input: string, expectedStatus: number, expectedMessage: string]; const cases: TestCase[] = [ ['valid@email.com', 200, 'Success'], ['', 400, 'Email required'], ];
14
What are optional properties and optional parameters? How do you define them?BeginnerP2
Basic Answer

Optional properties use ? after the property name: interface User { name: string; email?: string; }. Optional function parameters follow the same pattern: function createUser(name: string, age?: number). Optional parameters must come after required parameters. In test automation, optional properties are useful for API response types where certain fields may or may not be present.

15
What are readonly properties? When would you use them in test configuration?BeginnerP2
Basic Answer

readonly prevents property reassignment after initialization. It's not runtime enforcement — it's a compile-time check. In test automation, use readonly for configuration values, environment URLs, API endpoints, and constants that shouldn't be modified during test execution. Combined with interface and as const, it creates robust, immutable configuration objects.

interface TestConfig { readonly baseUrl: string; readonly timeout: number; readonly browser: 'chromium' | 'firefox'; testData?: Record<string, unknown>; // Non-readonly — can be modified per test }
16
What is optional chaining (?.) and the non-null assertion operator (!)? When would you use each?BeginnerP2
Basic Answer

Optional chaining (user?.address?.city) safely accesses nested properties — returns undefined if any intermediate property is null or undefined, instead of throwing. The non-null assertion (element!) tells TypeScript "this value is not null/undefined" — it's a compile-time only assertion with no runtime effect. Use optional chaining liberally for safe data access. Use non-null assertions sparingly — only when you've verified the value exists through other means (e.g., after an explicit check).

// Safe nested access const city = user?.address?.city; // undefined instead of crash // Non-null assertion — use cautiously const element = await page.$('.submit-btn'); await element!.click(); // Only safe if element is guaranteed to exist
17
How do enum work in TypeScript? Compare numeric enums, string enums, and const enums. Give a Playwright browser type example.BeginnerP2
Basic Answer

Numeric enums auto-increment from 0: enum Color { Red, Green, Blue }. String enums have explicit values: enum Browser { Chromium = 'chromium', Firefox = 'firefox', WebKit = 'webkit' }. Const enums (const enum) are inlined at compile time with no runtime overhead — they disappear from the compiled JavaScript. String enums are preferred for API contracts and test data because they're more descriptive and have better runtime behavior.

enum BrowserType { Chromium = 'chromium', Firefox = 'firefox', WebKit = 'webkit' } const config = { browser: BrowserType.Chromium, headless: true }; // config.browser is typed as BrowserType, not just string
19
What are literal types? Give a test automation example using string literal types.IntermediateP2
Basic Answer

Literal types are types that represent specific values rather than general categories: type Environment = 'dev' | 'staging' | 'prod'. Combined with union types, they provide type-safe configuration. TypeScript narrows to literal types when using as const.

type Browser = 'chromium' | 'firefox' | 'webkit'; type LogLevel = 'debug' | 'info' | 'warn' | 'error'; const config = { browser: 'chromium' as const, headless: true };
20
What is satisfies operator and when would you use it?AdvancedP2
Basic Answer

satisfies (TS 4.9+) validates that a value matches a type WITHOUT widening the value's type. This preserves the literal type for inference while ensuring type safety. Useful for test configuration objects where you want both validation and narrow inference.

const config = { browser: 'chromium', // TypeScript infers as string (wide) timeout: 30000 } satisfies Record<string, unknown>; // Without satisfies, config.browser was 'string' — now stays literal
21
What are tuple types and variadic tuple types?AdvancedP2
Basic Answer

Tuples are arrays with fixed length and known element types: [string, number]. Variadic tuple types (TS 4.0+) use spread with generics: [...T, ...U]. Useful for typed function parameters in Playwright test fixtures and generic utility functions.

22
How do you create type-safe Page Objects in Playwright using TypeScript?AdvancedP2
Basic Answer

Define interfaces for page contracts, implement classes with typed methods, and use generics for reusable component patterns. Leverage Playwright's built-in Locator type and custom return types for page methods.

interface LoginPage { readonly emailInput: Locator; readonly passwordInput: Locator; readonly submitButton: Locator; login(email: string, password: string): Promise<DashboardPage>; } class PlaywrightLoginPage implements LoginPage { constructor(private page: Page) {} readonly emailInput = this.page.getByLabel('Email'); readonly passwordInput = this.page.getByLabel('Password'); readonly submitButton = this.page.getByRole('button', { name: 'Login' }); async login(email: string, password: string): Promise<DashboardPage> { await this.emailInput.fill(email); await this.passwordInput.fill(password); await this.submitButton.click(); return new DashboardPage(this.page); } }
23
How does TypeScript handle async/await return types? What is the return type of an async function?IntermediateP2
Basic Answer

An async function always returns a Promise<T> where T is the type of the resolved value. If you return string from an async function, the return type is Promise<string>. TypeScript infers this automatically but you should annotate it explicitly for public APIs.

24
What is the never type and when does it occur?AdvancedP2
Basic Answer

never represents values that never occur. Functions that always throw or have infinite loops return never. It's also used in exhaustive type checking: if a switch statement covers all union members, the default case can assert the value is never. If a new union member is added but not handled, TypeScript flags it.

25
What is NonNullable<T> and how does it prevent null reference errors?IntermediateP2
Basic Answer

NonNullable<T> removes null and undefined from T. NonNullable<string | null | undefined> becomes string. Use it to ensure optional values are present after runtime validation.

26
How do decorators work in TypeScript? Give a Playwright-related use case.ExpertP2
Basic Answer

Decorators are functions that modify classes, methods, properties, or parameters. They use the @expression syntax and execute at design time. In Playwright, decorators can add logging, retry logic, or screenshot capture to page object methods declaratively.

27
How do you create a type-safe configuration system for test environments?AdvancedP2
Basic Answer

Use union types for environment names, intersection types for base+override configs, and utility types like Partial and DeepPartial for configuration merging. Use as const and satisfies for compile-time validation of configuration values.

P3 — Valuable Context

Less common but valuable for senior roles.

28
What are indexed access types? How do they differ from keyof?AdvancedP3
Basic Answer

keyof T returns the union of all property keys of T. Indexed access T[K] returns the type of property at key K. Combined: T[keyof T] returns the union of all property value types.

29
Explain Parameters<T> and ReturnType<T> utility types with examples.AdvancedP3
Basic Answer

Parameters<T> extracts function parameter types as a tuple. ReturnType<T> extracts the return type. Both use conditional types with infer. Useful for typing callback functions and wrapping third-party APIs.

30
How does declaration merging work with interfaces?AdvancedP3
Basic Answer

Interfaces with the same name in the same scope automatically merge their members. This is used for augmenting types from libraries (e.g., adding custom properties to Express Request). type aliases do not support declaration merging — they error on duplicate names.

31
What is brand/nominal typing in TypeScript? How do you implement it?ExpertP3
Basic Answer

TypeScript uses structural typing (duck typing). Brand types add a phantom property to create nominal-like type safety: type UserId = string & { readonly __brand: 'UserId' }. This prevents mixing up IDs of different entities even though they're all strings.

32
What are template literal types? Give a Playwright locator example.AdvancedP3
Basic Answer

Template literal types create types by string manipulation: type EventName = \`on\${Capitalize}\`. In Playwright, they can create type-safe dynamic locator patterns.

type TestId<T extends string> = `data-testid="${T}"`; type UserTestId = TestId<'user-name' | 'user-email'>; // "data-testid=\"user-name\" | data-testid=\"user-email\""
