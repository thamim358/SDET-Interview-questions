P1 — Frequently Asked Questions

Must-know for TypeScript-focused SDET interviews.

1
What are the advantages of TypeScript over JavaScript for large-scale test automation projects?IntermediateP1
Basic Answer

TypeScript adds static types to JavaScript — it catches errors at compile time instead of runtime. This is important for large test suites where changing code without types can cause hidden failures. Benefits: better IDE help (autocomplete, refactoring), self-documenting code, fewer runtime errors, easier onboarding. For Playwright, TypeScript gives you type-safe locators, page objects, and API response validation.

Real Interview Answer

"TypeScript catches bugs during compilation that would otherwise fail at runtime. For example, if an API response changes shape, TypeScript shows the error in ALL tests using that response type immediately. Combined with Playwright's type-safe locators, this reduces flaky tests from type mismatches."

2
Explain the difference between interface and type alias. When would you use each?AdvancedP1
Basic Answer

Both define object shapes, but they differ. interface can be merged (reopened to add more properties) and extends easily with extends. type can do unions, intersections, primitives, and tuples — more flexible for complex types. Use interface for public APIs and object shapes that might be extended. Use type for unions, intersections, complex types, or computed properties.

Detailed Answer
// Interface — extensible, declaration merging interface User { id: number; name: string; } interface User { email: string; } // Merged — User now has id, name, email interface AdminUser extends User { role: 'admin'; } // Type — flexible, can use unions/intersections type Status = 'active' | 'inactive' | 'pending'; type ApiResponse<T> = { data: T; error: null } | { data: null; error: string }; type StringOrNumber = string | number; type UserWithRole = User & { role: string }; // Intersection
Real Interview Answer

"In my Playwright framework, I use interfaces for page object contracts — each interface defines what methods a page has. I use types for API responses, discriminated unions for test results, and template literal types for dynamic locators. Rule of thumb: interface for objects others depend on, type for everything else."

3
What are generics? Implement a generic API response wrapper type.AdvancedP1
Basic Answer

Generics let you create reusable components that work with different types while keeping type safety. They use angle brackets <T> as a placeholder for the type. A generic API response wrapper ensures type safety across all API calls in a test framework.

Detailed Answer
interface ApiResponse<T> { status: number; message: string; data: T; timestamp: string; } interface PaginatedResponse<T> extends ApiResponse<T[]> { pagination: { page: number; totalPages: number; totalItems: number; }; } // Usage in test framework interface User { id: number; name: string; email: string; } async function getUser<T>(id: number): Promise<ApiResponse<T>> { const response = await request.get(`/api/users/${id}`); return response.json(); } // Type-safe — TypeScript knows user.data has id, name, email const user = await getUser<User>(1); expect(user.data.name).toBe('Alice');
4
What are union types and intersection types? Give real-world examples from test automation.IntermediateP1
Basic Answer

Union types (A | B) let a value be one of several types. Intersection types (A & B) combine multiple types into one. In testing: unions model API response states (success/failure), intersections combine config with environment variables.

type TestResult = { passed: true; duration: number } | { passed: false; error: string }; type Config = BrowserConfig & EnvironmentConfig & AuthConfig; type LocatorStrategy = 'role' | 'testid' | 'text' | 'css' | 'xpath';
5
How does type narrowing work? Explain typeof, instanceof, in, and discriminated unions.AdvancedP1
Basic Answer

Type narrowing is TypeScript's way of making a broad type more specific in certain parts of your code. typeof x === 'string' narrows to string. instanceof Error narrows to Error. 'property' in obj checks if a property exists. Discriminated unions use a literal property to narrow — the most powerful pattern for handling API responses.

type ApiState<T> = | { status: 'loading' } | { status: 'success'; data: T } | { status: 'error'; error: string }; function handleState<T>(state: ApiState<T>) { if (state.status === 'loading') return 'Loading...'; // Narrowed: { status: 'loading' } if (state.status === 'success') return state.data; // Narrowed: { status: 'success'; data: T } return state.error; // Narrowed: { status: 'error'; error: string } }
6
What are utility types? Explain Partial<T>, Required<T>, Pick<T>, Omit<T>, Record<K,T>, and Readonly<T>.IntermediateP1
Basic Answer

Utility types are built-in tools that transform types. Partial<T> — makes all properties optional (great for test data overrides). Required<T> — makes all required. Pick<T, K> — picks specific keys. Omit<T, K> — removes specific keys. Record<K,T> — creates an object type with keys K and values T. Readonly<T> — makes properties read-only.

interface User { id: number; name: string; email: string; role: string; } type PartialUser = Partial<User>; // All optional — for test data overrides type UserPreview = Pick<User, 'id' | 'name'>; // Just id and name type UserWithoutRole = Omit<User, 'role'>; // All except role type UserMap = Record<number, User>; // { [key: number]: User }
7
What is the difference between any and unknown? When would you use each?IntermediateP1
Basic Answer

any turns off ALL type checking — you can do anything with it. unknown is the safe version — you MUST narrow it before using it (via typeof, type guards, or assertions). NEVER use any in production code. Use unknown for data from outside your app (API responses, user input) where you don't know the exact shape.

⚠️ Interview Trap Saying "I use any sometimes to save time" is a red flag. Always prefer unknown and narrow it. If you use any, justify it clearly (e.g., "during a migration" or "for a third-party library with bad types").
8
What are mapped types? Create a custom mapped type.ExpertP1
Basic Answer

Mapped types loop over the keys of a type to create new types. Syntax: { [P in K]: T }. The built-in Partial, Readonly, Pick, and Record are all mapped types. You can create your own to transform property modifiers or values.

type Nullable<T> = { [K in keyof T]: T[K] | null }; type Stringify<T> = { [K in keyof T]: string }; type DeepReadonly<T> = { readonly [K in keyof T]: T[K] extends object ? DeepReadonly<T[K]> : T[K]; };
9
What are conditional types? Explain with the infer keyword.ExpertP1
Basic Answer

Conditional types pick one type or another based on a condition: T extends U ? X : Y. The infer keyword lets you pull out a type from inside another type. ReturnType<T> and Parameters<T> use infer internally.

type IsString<T> = T extends string ? 'yes' : 'no'; // Extract Promise value type type Unwrap<T> = T extends Promise<infer U> ? U : T; type Val = Unwrap<Promise<number>>; // number // Extract function return type type MyReturnType<T> = T extends (...args: any[]) => infer R ? R : never;
10
How do you type API responses in Playwright? Design a generic API response type.AdvancedP1
Basic Answer

Playwright's APIResponse provides generic methods. Create a typed wrapper that validates the response shape at compile time. Use discriminated unions for success/error states and utility types for pagination.

interface ApiClient { get<T>(url: string): Promise<{ data: T; status: number }>; post<T, B>(url: string, body: B): Promise<{ data: T; status: number }>; } class PlaywrightApiClient implements ApiClient { constructor(private request: typeof request) {} async get<T>(url: string) { const response = await this.request.get(url); const data: T = await response.json(); return { data, status: response.status() }; } }

P2 — Commonly Asked Questions

Beginner to intermediate questions frequently appearing in technical screens.

11
What are the basic types in TypeScript? How do you annotate variables and function parameters?BeginnerP2
Basic Answer

Basic types are: string, number, boolean, null, undefined, void, and any. You write let name: string = 'Alice' for variables and function greet(name: string): string { ... } for functions. TypeScript can also guess types (inference), so you don't always need to write them. In Playwright tests, annotating parameters ensures type-safe page object methods.

Real Interview Answer

"Basic type annotations catch bugs early. If a function expects string but gets number, TypeScript catches it at compile time before the test runs. In Playwright page objects, I annotate all method parameters and return types — this makes the framework self-documenting and prevents runtime type errors."

12
How does type inference work? When do you need explicit type annotations vs letting TS infer?BeginnerP2
Basic Answer

TypeScript guesses types from values. let count = 5 → inferred as number. Inference works well for simple cases. Use explicit types when: (1) a function could return multiple types, (2) defining complex object shapes (interfaces), (3) public API function parameters and return types, (4) when inference is too broad (e.g., let data = {} is just {}, not a specific shape). In Playwright, inference works for simple locator assignments, but explicit types are better for fixtures and page objects.

// TypeScript infers: string let name = 'Alice'; // Explicit annotation needed for complex types interface User { id: number; name: string; } const getUser = async (): Promise<User> => { ... }; // Number is too broad — use literal type let port: 3000 | 3001 | 3002 = 3000;
13
What are arrays and tuples in TypeScript? How do they differ?BeginnerP2
Basic Answer

Arrays have any length and all elements must be the same type: string[] or Array<number>. Tuples have a fixed length and each position has a specific type: [string, number]. Tuples check both position and type — [3, 'hello'] errors if the type is [string, number]. In tests, tuples are great for parameterized test cases where each parameter has a different type.

type TestCase = [input: string, expectedStatus: number, expectedMessage: string]; const cases: TestCase[] = [ ['valid@email.com', 200, 'Success'], ['', 400, 'Email required'], ];
14
What are optional properties and optional parameters? How do you define them?BeginnerP2
Basic Answer

Optional properties use ? after the name: interface User { name: string; email?: string; }. Same for function parameters: function createUser(name: string, age?: number). Optional parameters must come after required ones. In test automation, optional properties are useful for API response types where some fields might be missing.

15
What are readonly properties? When would you use them in test configuration?BeginnerP2
Basic Answer

readonly stops you from changing a property after it's set. It's a compile-time check, not runtime. In test automation, use readonly for config values, URLs, API endpoints, and constants that shouldn't change during tests. Combined with interface and as const, it creates safe, unchangeable configuration objects.

interface TestConfig { readonly baseUrl: string; readonly timeout: number; readonly browser: 'chromium' | 'firefox'; testData?: Record<string, unknown>; // Non-readonly — can be modified per test }
16
What is optional chaining (?.) and the non-null assertion operator (!)? When would you use each?BeginnerP2
Basic Answer

Optional chaining (user?.address?.city) safely accesses nested properties — returns undefined if anything along the path is null or undefined, instead of crashing. The non-null assertion (element!) tells TypeScript "trust me, this is not null" — it's compile-time only, no runtime effect. Use optional chaining freely. Use non-null assertions carefully — only when you're sure the value exists (like after checking).

// Safe nested access const city = user?.address?.city; // undefined instead of crash // Non-null assertion — use cautiously const element = await page.$('.submit-btn'); await element!.click(); // Only safe if element is guaranteed to exist
17
How do enum work in TypeScript? Compare numeric enums, string enums, and const enums. Give a Playwright browser type example.BeginnerP2
Basic Answer

Numeric enums start at 0 and auto-increment: enum Color { Red, Green, Blue }. String enums have specific values: enum Browser { Chromium = 'chromium', Firefox = 'firefox', WebKit = 'webkit' }. Const enums (const enum) are replaced at compile time with no runtime overhead — they vanish from the compiled code. String enums are better for API contracts and test data because they're clearer and work better at runtime.

enum BrowserType { Chromium = 'chromium', Firefox = 'firefox', WebKit = 'webkit' } const config = { browser: BrowserType.Chromium, headless: true }; // config.browser is typed as BrowserType, not just string
19
What are literal types? Give a test automation example using string literal types.IntermediateP2
Basic Answer

Literal types represent specific values instead of general categories: type Environment = 'dev' | 'staging' | 'prod'. With union types, they give you type-safe configuration. TypeScript narrows to literal types when you use as const.

type Browser = 'chromium' | 'firefox' | 'webkit'; type LogLevel = 'debug' | 'info' | 'warn' | 'error'; const config = { browser: 'chromium' as const, headless: true };
20
What is satisfies operator and when would you use it?AdvancedP2
Basic Answer

satisfies (TS 4.9+) checks that a value matches a type WITHOUT making the type wider. This keeps the literal type for inference while still checking for safety. Useful for test configuration objects where you want both validation and narrow types.

const config = { browser: 'chromium', // TypeScript infers as string (wide) timeout: 30000 } satisfies Record<string, unknown>; // Without satisfies, config.browser was 'string' — now stays literal
21
What are tuple types and variadic tuple types?AdvancedP2
Basic Answer

Tuples are arrays with fixed length and specific types per position: [string, number]. Variadic tuple types (TS 4.0+) use spread with generics: [...T, ...U]. Useful for typed function parameters in Playwright fixtures and generic utilities.

22
How do you create type-safe Page Objects in Playwright using TypeScript?AdvancedP2
Basic Answer

Define interfaces for page contracts, implement classes with typed methods, and use generics for reusable component patterns. Use Playwright's built-in Locator type and custom return types for page methods.

interface LoginPage { readonly emailInput: Locator; readonly passwordInput: Locator; readonly submitButton: Locator; login(email: string, password: string): Promise<DashboardPage>; } class PlaywrightLoginPage implements LoginPage { constructor(private page: Page) {} readonly emailInput = this.page.getByLabel('Email'); readonly passwordInput = this.page.getByLabel('Password'); readonly submitButton = this.page.getByRole('button', { name: 'Login' }); async login(email: string, password: string): Promise<DashboardPage> { await this.emailInput.fill(email); await this.passwordInput.fill(password); await this.submitButton.click(); return new DashboardPage(this.page); } }
23
How does TypeScript handle async/await return types? What is the return type of an async function?IntermediateP2
Basic Answer

An async function ALWAYS returns a Promise<T> — T is the type of the value you return. If you return string, the return type is Promise<string>. TypeScript infers this automatically, but you should write it explicitly for public APIs.

24
What is the never type and when does it occur?AdvancedP2
Basic Answer

never represents values that never happen. Functions that always throw errors or have infinite loops return never. Also used for exhaustive type checking: if a switch covers all union members, the default case can assert the value is never. If you add a new union member and forget to handle it, TypeScript catches it.

25
What is NonNullable<T> and how does it prevent null reference errors?IntermediateP2
Basic Answer

NonNullable<T> removes null and undefined from T. NonNullable<string | null | undefined> becomes just string. Use it to make sure optional values exist after runtime validation.

26
How do decorators work in TypeScript? Give a Playwright-related use case.ExpertP2
Basic Answer

Decorators are functions that modify classes, methods, properties, or parameters. They use the @expression syntax. In Playwright, decorators can add logging, retry logic, or screenshot capture to page object methods cleanly.

27
How do you create a type-safe configuration system for test environments?AdvancedP2
Basic Answer

Use union types for environment names, intersection types for base + override configs, and utility types like Partial and DeepPartial for merging configs. Use as const and satisfies for compile-time validation of config values.

P3 — Valuable Context

Less common but valuable for senior roles.

28
What are indexed access types? How do they differ from keyof?AdvancedP3
Basic Answer

keyof T gives you all the property names of T as a union. Indexed access T[K] gives you the type of property at key K. Combined: T[keyof T] gives you all the property value types as a union.

29
Explain Parameters<T> and ReturnType<T> utility types with examples.AdvancedP3
Basic Answer

Parameters<T> gets the parameter types of a function as a tuple. ReturnType<T> gets the return type. Both use conditional types with infer. Useful for typing callbacks and wrapping third-party APIs.

30
How does declaration merging work with interfaces?AdvancedP3
Basic Answer

Interfaces with the same name in the same scope automatically merge their members. Used for adding types to libraries (e.g., adding custom properties to Express Request). type aliases do NOT support merging — duplicate names cause an error.

31
What is brand/nominal typing in TypeScript? How do you implement it?ExpertP3
Basic Answer

TypeScript uses structural typing (duck typing — if it looks like a duck, it's a duck). Brand types add a fake property to create nominal-like type safety: type UserId = string & { readonly __brand: 'UserId' }. This stops you from mixing up different kinds of IDs even though they're all strings.

32
What are template literal types? Give a Playwright locator example.AdvancedP3
Basic Answer

Template literal types create types by manipulating strings: type EventName = \`on\${Capitalize}\`. In Playwright, they can create type-safe dynamic locator patterns.

type TestId<T extends string> = `data-testid="${T}"`; type UserTestId = TestId<'user-name' | 'user-email'>; // "data-testid=\"user-name\" | data-testid=\"user-email\""
